PROJECT OVERVIEW
────────────────────────────────────────────────────────────────────────────────
JWT Shield is an enterprise-grade security monitoring platform that provides 
real-time detection and prevention of JWT (JSON Web Token) attacks across 
multiple client applications. It operates as a centralized security-as-a-service 
platform where external applications report authentication events, and JWT Shield 
provides comprehensive threat intelligence, IP reputation tracking, and automated 
blocking mechanisms.

CORE ARCHITECTURE
────────────────────────────────────────────────────────────────────────────────

1. MULTI-TENANT ISOLATION
   - Each client application registers as a separate tenant
   - Unique API keys (format: jwtshield_live_<48-char-hex>) for authentication
   - Complete data isolation using Redis key prefixing (tenant:<id>:*)
   - Zero-break compatibility: works with or without tenant context

2. COMPONENTS
   ├─ Core Security Engine (@jwt-shield/core)
   │  └─ 5-stage validation pipeline for JWT verification
   ├─ Enterprise API (port 3001)
   │  ├─ /enterprise/register — tenant onboarding
   │  ├─ /enterprise/ingest — event ingestion (batch up to 100 events)
   │  └─ /api/stats/* — tenant-scoped analytics
   └─ Real-time Dashboard (port 5173)
      └─ Live WebSocket updates, tenant-filtered views

3. SECURITY PIPELINE (5 STAGES)
   Stage 1: Header Sanitization
            - Rejects dangerous fields: jku, jwk, x5u, x5c
            - Validates kid format (no path traversal, max 128 chars)
   
   Stage 2: Algorithm Pinning
            - Enforces RS256 only
            - Blocks "none" algorithm exploit
            - Prevents algorithm downgrade attacks
   
   Stage 3: Key Integrity
            - RSA signature verification
            - Public key validation
   
   Stage 4: Claims Verification
            - Issuer/audience validation
            - Expiration checking (with 30s clock skew tolerance)
            - Custom claim validation
   
   Stage 5: Replay Protection
            - JTI (JWT ID) tracking in Redis
            - Single-use token enforcement

ATTACK DETECTION CAPABILITIES
────────────────────────────────────────────────────────────────────────────────

DETECTED ATTACK VECTORS:
├─ alg_none              — Algorithm "none" bypass attempt
├─ algorithm_confusion   — Algorithm downgrade (HS256 → RS256 confusion)
├─ kid_injection         — Path traversal in kid header field
├─ header_injection      — Dangerous fields (jku, jwk, x5u, x5c)
├─ replay_attack         — Token reuse detection via JTI tracking
├─ invalid_signature     — Signature verification failure
├─ malformed_token       — Broken JWT structure
├─ invalid_claims        — Expired or invalid claims
└─ bad_credentials       — Failed login attempts

IP REPUTATION & BLOCKING SYSTEM
────────────────────────────────────────────────────────────────────────────────

SLIDING WINDOW ALGORITHM:
- Rejection Window: 5 minutes (300s) — resets on every failed request
- Attack Window: 1 hour (3600s) — resets on every detected attack
- Catches slow, distributed attacks that fixed windows miss

THRESHOLDS (configurable via .env):
├─ IP_WARN_THRESHOLD: 5 rejections → warning flag
├─ IP_BLOCK_THRESHOLD: 10 rejections → temporary block (1 hour default)
└─ IP_ATTACK_PERMANENT_THRESHOLD: 3 attacks → permanent ban

BLOCKING BEHAVIOR:
- Temporary blocks expire after IP_BLOCK_DURATION_SECONDS (default: 3600s)
- Permanent blocks persist until manual unblock via dashboard
- Blocked IPs return 403 Forbidden on all subsequent requests

API CALL ANALYSIS & COST ESTIMATION
────────────────────────────────────────────────────────────────────────────────

OPERATION BREAKDOWN (per request):

1. EVENT INGESTION (/enterprise/ingest)
   ├─ Tenant lookup: 1 Redis GET
   ├─ Stats update: 2 Redis INCR (total_requests, attack_vector)
   ├─ IP reputation check: 2 Redis GET (perm, block)
   ├─ IP counter update: 1 Redis INCR + 1 Redis EXPIRE
   ├─ Attack logging: 1 Redis LPUSH + 1 Redis LTRIM
   └─ TOTAL: 8 Redis operations per event

2. DASHBOARD STATS (/api/stats/summary)
   ├─ Uptime: 1 Redis GET
   ├─ Counters: 2 Redis GET (total_requests, total_blocked)
   ├─ Attack breakdown: 1 Redis KEYS + N Redis GET (N = attack types)
   ├─ Blocked IPs: 1 Redis KEYS
   └─ TOTAL: 5 + N Redis operations (N typically 5-10)

3. EVENT LOG (/api/stats/events)
   └─ 1 Redis LRANGE (retrieves last 100 events)

4. BLOCKED IP LIST (/api/stats/blocked-ips)
   ├─ Block keys: 1 Redis KEYS + N Redis GET + N Redis TTL
   ├─ Perm keys: 1 Redis KEYS + M Redis GET
   └─ TOTAL: 2 + 2N + M Redis operations

5. IP UNBLOCK (/api/stats/unblock-ip)
   └─ 1 Redis DEL (removes 4 keys: block, perm, rej, atk)

INTEGRATION GUIDE FOR CLIENT APPLICATIONS
────────────────────────────────────────────────────────────────────────────────

STEP 1: REGISTER AS TENANT
curl -X POST http://localhost:3001/enterprise/register \
  -H "Content-Type: application/json" \
  -d '{"name": "YourAppName"}'

Response: { "apiKey": "jwtshield_live_...", "tenantId": "tenant_...", ... }

STEP 2: ADD EVENT REPORTER TO YOUR APP
const SHIELD_URL = 'http://localhost:3001/enterprise/ingest';
const SHIELD_KEY = 'jwtshield_live_...';
let eventBatch = [];

setInterval(() => {
  if (eventBatch.length === 0) return;
  fetch(SHIELD_URL, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Shield-Key': SHIELD_KEY
    },
    body: JSON.stringify({ events: eventBatch.splice(0, 100) })
  });
}, 1000);

STEP 3: REPORT EVENTS FROM YOUR AUTH MIDDLEWARE
function reportEvent(req, token, blocked, attackVector, detail) {
  eventBatch.push({
    timestamp: new Date().toISOString(),
    source_ip: req.headers['x-forwarded-for']?.split(',')[0] || req.ip,
    user_agent: req.headers['user-agent'],
    token_fingerprint: sha256(token.slice(0, 20)),
    blocked,
    attack_vector: attackVector,
    detail
  });
}
